Gate 05I-C — Low-Voltage Communications & Power State Integrity
This gate establishes the complete verification protocol for the low-voltage communication buses (CAN_2, CAN_3, and diagnostic/display lines) and low-power state transitions in an isolated, non-powered-HV bench environment.
These protocols ensure that before any controller is integrated into a physical vehicle, its communication stacks, frame-handling engines, transceiver configurations, and sleep/wake controllers are robust against electrical, logical, and low-power fault states.

05I-C1 — Communication Network Integrity
This subgate validates that the physical, protocol, and application layers of the communication buses function correctly under stress, reject malformed or invalid data, and maintain absolute isolation from the production vehicle network (CAN_1).
1. Physical & Protocol Layer Integrity
At this boundary, the physical transceivers and microcontroller-level CAN controllers must automatically filter out corrupted physical frames before they reach the application layer.
CAN_1 Passive Listen-Only Isolation: The VCU CAN_1 controller must be configured to never transmit frames, assert TXD dominant, participate in ACK, or emit active error frames. Verification Method: Because other OEM nodes on a mixed bench network may acknowledge frames, validation cannot rely solely on decoded bus logs. A high-bandwidth oscilloscope probe must be placed directly on the VCU physical TXD pin and transceiver path to verify it remains at Vcc (recessive high) under all operating conditions.
Physical & Protocol Fault Filtering: Physical-layer faults—such as bad CRC, bit-stuffing errors, form errors, and bus-level bit errors—must be handled and discarded directly by the CAN controller hardware. These faults must not cause microcontroller register lockups, buffer overflows, or application-layer crashes.
2. Application Layer Validation
Once a physically valid frame passes the CAN controller hardware, the application-layer software must validate its logical contents against strict structural and safety boundaries.
DBC Version Matching & Build Engine Verification: To prevent the classic failure of silent, mismatched database decoding:
The expected DBC version hash/checksum must be stored directly in the Build Engine.
The VCU firmware build must explicitly declare its expected DBC version.
The bench testing tools must log the actual DBC version used during the run.
Any detected version hash mismatch must trigger a BENCH_HARD_BLOCK_PENDING_REVIEW state.
Data Integrity Protections: Critical signals must use safety wrappers including Rolling Alive Counters (detects dropped/frozen frames) and Application-Layer Checksums (detects payload corruption that bypasses physical CRCs). Mismatches in either wrapper must immediately invalidate the frame.
Out-of-Range Signal Scaling: Any decoded physical signal that exceeds defined physical limits (e.g., cell voltages, motor RPM, pedal position) must be rejected by the application parser, flag a diagnostic trouble code (DTC), and safely transition the state machine to a zero-torque or safe state.

05I-C2 — Sleep / Wake / Parasitic Drain
This subgate validates the low-voltage power state engine, ensuring the system can reliably enter low-power sleep to prevent parasitic 12V battery drain, recover from brownout conditions, and wake only on authorized triggers without leaking noise or wake commands to the vehicle networks.
1. Power State Definitions & Current Budgets
To ensure the bench system does not drain the 12V battery during long-term storage, current budgets must be tracked per node and as a total aggregated system.
Sleep-Current Targets:
VCU_sleep_current_target <= 1.0 mA
BMS_logic_sleep_current_target <= 1.5 mA
Inverter_logic_sleep_current_target <= 1.0 mA
Display_sleep_current_target <= 0.5 mA
Total_system_sleep_current_target <= 4.0 mA (Aggregated total bench quiescent draw)
2. Transition Limits & Wakeup Rules
Sleep Transition Window: The transition from key-off/ignition-off to absolute sleep current (<= 4.0 mA total system) must be complete within <= 2.0 seconds.
Stuck-Awake Protection: If any node fails to transition to sleep within 5.0 seconds of the command (e.g., due to a stuck transceiver, bus babbling, or firmware state lock), the VCU must flag a "Stuck Awake" fault and log the offending node before forcing local low-power entry if possible.
Brownout Recovery: If the low-voltage logic rail dips down to a brownout threshold (6.0V to 8.0V), the VCU must gracefully store non-volatile data, suspend active tasks, and recover to an aligned initialization state without spurious fault trips or stuck transceivers when voltage is restored to >= 9.0V.

Next Subgate: Gate 05I-D
Once the communications network integrity (05I-C1) and power/sleep states (05I-C2) are physically verified on the bench, the system transitions to the final bench-testing phase prior to vehicle physical commissioning:
Gate 05I-D: Low-Voltage End-to-End Bench Run / Integrated Faults.
[Dynamic Control Loops]: Driver Inputs & Shifter; Brake Override (Brake/Gas); Live Display Warn/Alert; Diagnostic Access Locks.
[Integrated Fault Cascades]: E-Stop / Inertia Switch; HVIL Open Circuit; BMS Over-Temp & Faults; Inverter Failure Cascades; CAN Heartbeat Loss (Off-nominal).
In Gate 05I-D, all components (VCU, BMS logic, inverter logic, display, and simulator interfaces) run dynamically in real-time. This subgate subjects the fully integrated low-voltage architecture to a series of coordinated, off-nominal fault cascades (such as opening the HVIL during an active torque command or triggering a sudden BMS fault) to verify that the integrated software logic handles failures safely and predictably.
